Criando um tutorial

Crie um novo tutorial que oriente os usuários na criação, execução e teste de uma aplicação Quarkus que utilize anotações de uma extensão imaginária.

Pré-requisitos

  • Cerca de 15 minutos

  • Um editor ou IDE que fornece realce de sintaxe e pré-visualizações para AsciiDoc, nativamente ou usando um plugin.

  • Já deve estar familiarizado com a visão geral do que é um Tutorial.

  • Tenha à mão as diretrizes de estilo e conteúdo do Quarkus como referência para a sintaxe necessária e outras convenções.

1. Decidir o título e o nome do arquivo

  1. Escolha um título para o tutorial. Para este exemplo prático, usaremos Serve Http requests using the Acme extension.

  2. Decida o nome do arquivo. Para este exemplo, usaremos acme-serve-http-requests-tutorial.adoc :

    • acme- este guia será agrupado com outros recursos relacionados com a extensão acme

    • serve-http-requests é um derivado do título do documento

    • -tutorial indicará que este documento é um tutorial (opcional)

    • .adoc para indicar que se trata de um arquivo asciidoc

2. Copiar o modelo de tutorial

Copie docs/src/main/diataxis/_templates/template-tutorial.adoc do repositório Quarkus para um novo arquivo chamado acme-serve-http-requests-tutorial.adoc.

3. Atualizar o cabeçalho do documento

[id="acme-serve-http-requests-tutorial"] (1)
= Serve Http requests using the Acme extension (2)
include::_attributes.adoc[] (3)
:categories: web (4)
:extension-status: experimental (5)
1 Utilize o nome do arquivo como uma identificação única para a seção.
2 Adicione o título como um cabeçalho de nível superior.
3 Incluir atributos adicionais que ajudem a definir uma formatação consistente e proporcionem a portabilidade da fonte.
4 Especifique a categoria web, uma vez que a nossa extensão imaginária funciona com pedidos Http.
5 A nossa extensão imaginária acme é experimental, por isso incluímos a declaração de estado da extensão no cabeçalho.
Pré-visualização do documento

A prévia do seu documento deve conter o título escolhido como um estilocabeçalho no topo da página.

4. Adicionar um resumo

Queremos que os leitores do nosso tutorial saibam o que eles conseguirão ao seguir as etapas do tutorial. Esse resumo deve ser conciso e usar verbos apropriados (criar, construir, implementar, …​) para definir expectativas. Os leitores devem ser capazes de determinar se querem se envolver com o conteúdo.

Create an application that uses unique annotations from the experimental acme extension to define two endpoints:
a simple HTTP endpoint and an endpoint that emits server-sent events (SSE).
We will also use Quarkus dev mode for iterative development and testing.
Pré-visualização do documento

A pré-visualização do seu documento deve conter o resumo como um parágrafo imediatamente a seguir ao cabeçalho.

Como a extensão acme é experimental, incluiremos {includes}/extension-status.adoc que fornece o texto de status da extensão. Ele usa o atributo de status da extensão definido no cabeçalho.

7.2. Usando um arquivo de origem

Neste tutorial, vamos incluir o código de um arquivo Java separado usando tags.

Considere este exemplo como uma aspiração. O código-fonte é mais comumente incluído diretamente no bloco de código-fonte. Embora haja vantagens em incluir o código-fonte dos arquivos Java, há detalhes que precisamos resolver em relação a onde o código referenciado deve ficar. Ajuda e ideias são bem-vindas!

Vamos criar um arquivo chamado acme-serve-http-requests-MyAcmeApplication.java.Embora esse não seja um nome de arquivo Java válido, ele fica bem ao lado da fonte do nosso tutorial.Vamos confiar na magia do IDE para a verificação da sintaxe.

Vamos adicionar o seguinte código a esse arquivo:

// tag::application[]
package org.acme;

import jakarta.enterprise.context.ApplicationScoped;

import org.acme.corp.Anvil;
import org.acme.corp.Toaster;

import io.smallrye.mutiny.Multi;

@ApplicationScoped // <1>
public class MyAcmeApplication {

    @Anvil(optional = {"name"}) // <2>
    public String hello(String name) {
        return String.format("Hello, %s!", (name == null ? "World" : name));
    }

    // tag::goodbye[]
    @Toaster // <1>
    public Multi<String> longGoodbye(String name) {
        return Multi.createFrom().items("Goodbye", ",", "Sweet", "Planet", "!"); // <2>
    }
    // end::goodbye[]
}
// end::application[]

Há alguns aspectos a ter em conta neste exemplo de código:

  • As chamadas do AsciiDoc são apresentadas como comentários em algumas linhas e a sua numeração não é sequencial.

  • As regiões de conteúdo do AsciiDoc são definidas por pares de etiquetas ( tag:: e end::) em comentários que envolvem diferentes seções de código.

7.3.2. Explorar o desenvolvimento e o teste contínuos

Agora é hora de orientar o usuário sobre como iniciar o modo de desenvolvimento do Quarkus. Um include comum faz a maior parte do trabalho para nós:

== Dev Mode: Hello, World!

Let's run our application using Quarkus' iterative development mode:

include::{includes}/devtools/dev.adoc[]

Once the console output from dev mode indicates that things are ready, use `curl` to invoke the `hello` endpoint:

[source,shell]
----
$ curl -w "\n" http://localhost:8080/hello
Hello, World!
----

Pass the name as a parameter:

[source,bash,subs=attributes+]
----
$ curl localhost:8080/hello -d '{"name": "bananas"}'
Hello, bananas!
----

You can leave dev mode running throughout the rest of the tutorial for continuous feedback as you make changes to code.
Use `CTRL-C` to exit dev mode.

Estamos fornecendo algumas coisas aqui:

  • Estamos utilizando um texto comum que descreve como iniciar o modo de desenvolvimento.

  • Descrevemos como utilizar curl para testar a saída do nosso ponto final definido.

Pré-visualização do documento

Your preview should contain a new section, "3. Dev Mode: Hello, World!" That section should include three methods of launching Quarkus dev mode (cli, maven, gradle). It should have codeblocks containing curl console commands and output, and instructions for how to exit dev mode.
